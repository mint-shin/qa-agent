7일/종강일 만료 계산

파일: uphone3_api .../service/WeeklyAttendanceService.java (약 414-428)

private LocalDate getRewardExpiredAt(Long memberSeq) {
    LocalDate defaultExpiredAt = LocalDate.now().plusDays(6); // +6일
    Purchase purchase = purchaseRepository.findInClassPurchase(memberSeq).orElse(null);
    if (purchase == null) return defaultExpiredAt;
    boolean hasNextPurchase = purchaseRepository.findNextPurchase(memberSeq).isPresent();
    if (!hasNextPurchase && purchase.getEndlessonDate() != null
        && purchase.getEndlessonDate().isBefore(defaultExpiredAt)) {
        return purchase.getEndlessonDate();
    }
    return defaultExpiredAt;
}

검증: 종강일이 7일 이내면 종강일 기준으로 보정 → 기획서 부합.
미구현: (1) 정기 상품(구독 자동갱신)인 경우 연말까지 간주하는 로직 없음. (2) "plusDays(6)"이 D-7 vs D-6 해석 차이 유발.

확률 추첨 (가중치 기반 랜덤)

파일: WeeklyAttendanceService.java (약 250-269)

private WeeklyAttendanceSetting pickSetting(List<WeeklyAttendanceSetting> settings) {
    double totalWeight = settings.stream()
        .mapToDouble(s -> s.getProbability() != null ? s.getProbability() : 0).sum();
    if (totalWeight <= 0) return settings.get(0);
    double roll = random.nextDouble() * totalWeight;
    double cumulative = 0;
    for (WeeklyAttendanceSetting s : settings) {
        cumulative += (s.getProbability() != null ? s.getProbability() : 0);
        if (roll < cumulative) return s;
    }
    return settings.get(settings.size() - 1);
}

잠재 이슈: java.util.Random을 공유 필드로 사용 시 멀티스레드 경합 가능. ThreadLocalRandom 또는 SecureRandom 권장.

3. 잠재 버그·결함 상세 (풀스택)

3.1 Critical

C-1. 리워드 등급 체계 3단계 vs 7단계 불일치

영향 범위: 프론트 GRADE_STYLES, 백엔드 WeeklyRewardGrade enum, LMS 관리자 세팅 UI 모두 S~F 기준.

기획서: "등급(상/중/하)에 따라 결과 화면 리워드 카드 컬러 차등."

권고: (A) 기획서를 S~F로 업데이트하거나 (B) enum/UI/DB를 3단계로 마이그레이션. 기존 출시 상태라면 (A)가 현실적. 사용자 노출 등급 텍스트는 그룹핑(S/A=상, B~D=중, E/F=하) 고려.

C-2. 정기 상품 종강 주차 분기 미구현

기획서 26.3.10: "정기 상품: 연도의 마지막 주차까지 모두 표시 / 일반: 종강 주차까지."

코드 현황: 백엔드 getRewardExpiredAt 및 weeklyList 생성 로직에 PurchaseType.REGULAR/ONESHOT 등 구분 분기 확인 불가.

리스크: 정기 구독자가 리포트에서 남은 주차가 전체 연말까지 표시되지 않고 현재 수강권의 endlessonDate에 따라 끊길 수 있음.

권고: WeeklyAttendanceService.buildWeeklyList 단계에서 구독형 여부를 확인해 연말까지 가상 주차 노드를 생성.

C-3. 자동화 테스트 커버리지 전무

프론트: Playwright e2e에 b2b2c-email-signup.spec.ts 외 AttendanceReport/RewardClaim 시나리오 없음.
백엔드: uphone3_api/src/test 하위 WeeklyAttendance* 테스트 0건.
리스크: 만료일 계산, 재수강 이벤트, 확률 추첨 등 복잡 로직이 회귀 테스트 없이 유지되며, 향후 수정 시 기존 동작 보장 불가.

3.2 Warning

W-1. 확률 추첨 동시성 (java.util.Random 사용)

여러 유저가 동시에 claim을 호출하면 Random 내부 seed 경합으로 성능 저하 및 예측 가능성 이슈. 보상 가치가 큰 경우 SecureRandom 사용 권장.

권고: private final Random random = new Random(); → ThreadLocalRandom.current() 또는 SecureRandom으로 교체.

W-2. 백엔드 리워드 대상자(B2C/KR) 차단 부재

claimWeeklyReward에서 회원 구분(B2C/B2B2C/B2B)과 국가 조합 검증 로직 확인되지 않음. 프론트 조건부 렌더에만 의존 시, 비대상자가 API를 직접 호출해 리워드 수령 시도 가능.

권고: 서비스 진입부에 if (!isRewardEligible(memberUphone)) throw new CustomException(...).

W-3. expiredAt = +6일 해석 애매함

"플러스 6일"은 오늘을 포함해 7일. 그러나 D-day 기준으로 "D-7, D-6..."으로 내려가며, 달성 당일의 잔여일이 D-6인지 D-7인지 해석에 차이가 있음. QA 테스트 시 실제 달성 직후 노출되는 D-day를 확인 필요.

W-4. 이벤트 기반 재설정의 과도기 레이스

@Async @TransactionalEventListener는 수강권 insert 완료 후 비동기 처리. 리포트 화면을 그 사이 새로고침하면 갱신 전 만료일이 표시될 수 있음.

권고: 이벤트 처리 후 WebSocket/SSE로 갱신 통지하거나, 프론트에서 구매 직후 일정 시간 delay + refetch.

W-5. 한글 에러 메시지 하드코딩 (글로벌 대응)

throw new CustomException("주간 목표를 달성하지 못했습니다.")처럼 한국어 문자열이 그대로 응답. 글로벌 B2C가 아닌 B2B2C 글로벌 회원에게는 부자연스러울 수 있음.

권고: 에러 코드(WEEKLY_NOT_ACHIEVED 등)만 반환하고 메시지는 프론트에서 i18n 처리.

W-6. 프론트 상수/연도 하드코딩 재확인

LearningReportProgressBar.tsx의 calendarYear >= 2026, RewardClaimPage의 dist < 50 등 v1에서 지적한 하드코딩 유지.

5. E2E / 통합 테스트 시나리오

시나리오 A. 풀스택 해피패스 (B2C/KR)

  1. 백엔드 상태 setup: member=B2C/KR, 당 주차 attendCount=goal (isAchieved=true, isClaimed=false, expiredAt=오늘+6일)
  2. 프론트 /AttendanceReport 진입 → reward-ready 노드 D-6 확인
  3. [리워드 받기] 클릭 → /RewardClaim 이동
  4. 드래그 + 스냅 → POST /weekly-attendance/{seq}/claim
  5. 서버 응답 rewardType/Grade/Amount 확인 → 결과 화면 렌더
  6. DB: MemberWeeklyAttendance.isClaimed=true, 적립금/쿠폰 테이블 insert 확인
  7. 복귀 시 finished-reward 전환, 기록보기 Red Dot 제거

시나리오 B. B2B 회원 차단 (정책 방어)

  1. B2B 회원 계정 로그인
  2. 프론트에서 리워드 UI 비노출 확인
  3. curl/Postman으로 /weekly-attendance/{seq}/claim 직접 호출
  4. 서버가 권한/정책 오류로 차단하는지 확인 (현재 코드상 미구현 의심 → 테스트 fail 시 Warning W-2 검증됨)

시나리오 C. 재수강 시 만료 원복

  1. 기존 수강권 종료(endlessonDate 임박) 상태에서 당 주차 달성
  2. expiredAt = 종강일 확인
  3. 신규 수강권 결제 → Purchase.schedulestatusCode=WAITING insert
  4. WeeklyAttendanceRewardExpireResetEvent 발행 로그 확인
  5. 프론트 refetch → expiredAt이 +6일로 원복 확인

시나리오 D. 종강 전 정기 상품 사용자

  1. 정기 구독 회원, 자동 갱신 활성
  2. AttendanceReport 진입 → 12월 마지막 주까지 노드가 모두 렌더되어야 함
  3. 실제 렌더 결과가 현재 수강권 endlessonDate에서 끊기면 Critical C-2 검증

시나리오 E. 확률 분포 검증

  1. LMS에서 S=10, A=20, B=30, C=20, D=10, E=5, F=5 로 설정
  2. 스테이징 회원 1만명으로 claim 스크립트 일괄 실행
  3. 결과 분포 집계 → 입력 확률 ±2% 이내 수렴 확인

6.4 결론

전체 스택을 교차검증한 결과, AttendanceReport/리워드 기능은 핵심 사용자 여정과 데이터 계약은 정합성이 매우 높습니다. 특히 7일 만료, 종강일 우선, 재수강 이벤트 재설정, 중복 수령 방어 등 v1에서 "의심"으로 표시된 기획 요구사항들이 실제 백엔드 코드에서 모두 확인되었습니다.

다만 (1) 리워드 등급 체계 표준화, (2) 정기/일반 상품 분기 구현, (3) 자동화 테스트 부재 세 가지는 배포 전 반드시 해결되어야 하며, 백엔드 레벨에서 대상자 차단이 부재한 점은 보안·정책 측면에서 우선순위가 높습니다.

본 리포트의 테스트 케이스 TC-01~TC-35를 QA 스프린트에 반영하고, 시나리오 A~E를 자동화하면 서비스 출시 안정성이 크게 향상됩니다.
